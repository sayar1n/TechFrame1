# Проверка пользовательских историй (User Stories) - Критерии приёмки

## 1. Введение

Данный документ описывает подход к проверке пользовательских историй для системы управления дефектами. Цель — убедиться, что реализованный функционал соответствует ожиданиям пользователей и бизнес-требованиям, зафиксированным на этапе анализа.

## 2. Подход к проверке

Проверка пользовательских историй будет осуществляться путем выполнения сценариев, основанных на каждой User Story, и сравнения полученных результатов с ожидаемыми. Это может включать как ручное тестирование, так и автоматизированные сквозные тесты (при наличии).

## 3. Критерии приёмки (Acceptance Criteria)

Для каждой пользовательской истории определены критерии приёмки, которые должны быть выполнены, чтобы считать историю реализованной и принятой. Общие критерии включают:

*   **Функциональность:** Система выполняет требуемое действие корректно.
*   **Удобство использования:** Интерфейс интуитивно понятен и легок в освоении.
*   **Производительность:** Действия выполняются в пределах допустимого времени отклика.
*   **Безопасность:** Доступ к функциям и данным соответствует ролевой модели.
*   **Валидация:** Ввод пользователя правильно валидируется, ошибки отображаются ясно.
*   **Соответствие макетам:** Внешний вид и расположение элементов соответствуют прототипам (если применимо).

## 4. Примеры сценариев приёмки по User Stories

Ниже приведены примеры User Stories и соответствующие им сценарии приёмки. Полный список User Stories находится в `docs/analysis/user_stories.md`.

### Пример 1: User Story (FT-001) - Регистрация пользователя

*   **User Story:** Как новый пользователь, я хочу зарегистрироваться в системе, чтобы получить доступ к функциям управления дефектами.
*   **Сценарий приёмки:**
    1.  Пользователь открывает страницу регистрации.
    2.  Пользователь вводит уникальное имя пользователя, действительный адрес электронной почты и пароль (минимум 8 символов, содержащий буквы и цифры).
    3.  Пользователь подтверждает пароль.
    4.  Пользователь нажимает кнопку "Зарегистрироваться".
    5.  Система успешно регистрирует пользователя и перенаправляет его на страницу входа.
    6.  (Альтернативный поток): Если имя пользователя или email уже заняты, система отображает соответствующее сообщение об ошибке.
    7.  (Альтернативный поток): Если пароли не совпадают, система отображает сообщение об ошибке.

### Пример 2: User Story (FT-002) - Вход в систему

*   **User Story:** Как зарегистрированный пользователь, я хочу войти в систему, чтобы получить доступ к своим проектам и дефектам.
*   **Сценарий приёмки:**
    1.  Пользователь открывает страницу входа.
    2.  Пользователь вводит зарегистрированное имя пользователя и пароль.
    3.  Пользователь нажимает кнопку "Войти".
    4.  Система успешно аутентифицирует пользователя и перенаправляет его на дашборд.
    5.  (Альтернативный поток): Если учетные данные неверны, система отображает сообщение об ошибке.

### Пример 3: User Story (FT-004) - Создание нового проекта (Менеджер)

*   **User Story:** Как менеджер, я хочу создать новый проект, чтобы организовать работу над конкретным строительным объектом.
*   **Сценарий приёмки:**
    1.  Менеджер входит в систему.
    2.  Менеджер переходит на страницу управления проектами или дашборд.
    3.  Менеджер нажимает кнопку "Создать проект".
    4.  Менеджер вводит название проекта (обязательно) и опциональное описание.
    5.  Менеджер нажимает кнопку "Сохранить".
    6.  Система создает новый проект и отображает его в списке проектов.
    7.  (Альтернативный поток): Если название проекта не указано, система отображает сообщение об ошибке.

### Пример 4: User Story (FT-005) - Создание дефекта (Инженер)

*   **User Story:** Как инженер, я хочу зарегистрировать новый дефект, чтобы зафиксировать проблему на строительном объекте.
*   **Сценарий приёмки:**
    1.  Инженер входит в систему.
    2.  Инженер переходит на страницу деталей проекта, для которого нужно создать дефект.
    3.  Инженер нажимает кнопку "Добавить дефект".
    4.  Инженер вводит заголовок дефекта (обязательно), описание, выбирает приоритет, статус, опционально назначает исполнителя и срок выполнения.
    5.  Инженер нажимает кнопку "Сохранить".
    6.  Система создает новый дефект и отображает его в списке дефектов проекта.
    7.  (Альтернативный поток): Если заголовок дефекта не указан, система отображает сообщение об ошибке.

## 5. Ответственность

Ответственность за проведение приемочного тестирования возлагается на тестировщиков и/или менеджеров проекта, которые будут сверять реализованный функционал с утвержденными пользовательскими историями.

## 6. Результаты

По результатам приемочного тестирования будет составлен отчет о соответствии функционала User Stories, выявленных несоответствиях и предложенных улучшениях.
