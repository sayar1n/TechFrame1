# План тестирования

## 1. Общие принципы тестирования

Целью тестирования является обеспечение качества, надежности и соответствия системы "Управление Дефектами" заявленным требованиям. Тестирование будет проводиться на различных уровнях, включая модульное, интеграционное и системное тестирование.

### Подходы:

*   **Тестирование по "белому ящику" (White-box testing):** Применяется для модульного тестирования бэкенда, чтобы убедиться в корректности внутренней логики и алгоритмов.
*   **Тестирование по "черному ящику" (Black-box testing):** Используется для проверки функциональных требований системы через её интерфейсы, без знания внутренней структуры.
*   **Регрессионное тестирование:** Проводится после каждого значимого изменения в коде для подтверждения того, что ранее работающие функции не были нарушены.

## 2. Уровни тестирования

### 2.1 Модульное тестирование (Unit Testing)

*   **Цель:** Проверка корректности отдельных функций, методов и классов на бэкенде.
*   **Инструменты:** `pytest` (Python).
*   **Объекты тестирования:** Функции в `backend/crud.py`, утилиты для хеширования паролей в `backend/main.py`.
*   **Покрытие:** Стремиться к высокому покрытию кода для критически важных модулей.
*   **Критерии завершения:** Все модульные тесты проходят успешно. Реализовано ≥ 5 юнит-тестов.

### 2.2 Интеграционное тестирование (Integration Testing)

*   **Цель:** Проверка взаимодействия между различными модулями системы (например, Frontend и Backend, Backend и Database).
*   **Инструменты:** `pytest` с `httpx` или `FastAPI TestClient` для бэкенда; `Playwright` или `Cypress` для фронтенда (для сквозных сценариев).
*   **Объекты тестирования:** API-эндпоинты (проверка цепочек запросов), сохранение и извлечение данных из БД.
*   **Критерии завершения:** Все интеграционные тесты проходят успешно. Реализовано ≥ 2 интеграционных сценария.

### 2.3 Системное тестирование (System Testing)

*   **Цель:** Проверка всей системы в целом на соответствие функциональным и нефункциональным требованиям, описанным в SRS и User Stories.
*   **Инструменты:** Ручное тестирование, автоматизированные сквозные тесты (с использованием `Playwright`/`Cypress`).
*   **Объекты тестирования:** Полный пользовательский флоу, включая аутентификацию, создание/управление проектами и дефектами, генерацию отчетов.
*   **Критерии завершения:** Все User Stories подтверждены, система соответствует нефункциональным требованиям.

### 2.4 Нагрузочное тестирование (Load Testing)

*   **Цель:** Оценка производительности системы под высокой нагрузкой для выявления узких мест и проверки соответствия требованиям по времени отклика.
*   **Инструменты:** `Locust`, `JMeter`.
*   **Сценарии:** Имитация 50 активных пользователей, выполняющих типичные операции (вход, просмотр проектов, создание дефектов).
*   **Критерии завершения:** Время отклика страниц ≤ 1 секунды для 50 активных пользователей.

## 3. Критерии приёмки

*   Все функциональные требования, описанные в SRS, реализованы и проверены.
*   Все нефункциональные требования (производительность, безопасность, удобство использования) выполнены.
*   Все модульные и интеграционные тесты успешно пройдены.
*   Пройдено нагрузочное тестирование с заданными параметрами.
*   Код имеет достаточное тестовое покрытие (будет отслеживаться с помощью инструментов покрытия кода).

## 4. Контроль покрытия кода тестами

*   Будет использоваться `coverage.py` для отслеживания покрытия кода модульными и интеграционными тестами.
*   Цель: Поддерживать покрытие на уровне не менее 80% для критически важных модулей бэкенда и фронтенда.
